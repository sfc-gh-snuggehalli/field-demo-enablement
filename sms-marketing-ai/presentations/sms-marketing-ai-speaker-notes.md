# Speaker Notes: Semantic Views & the AI-BI Stack on Snowflake

## Scenario Summary

The demo is grounded in an SMS/MMS marketing platform used by e-commerce brands. Shoppers
opt in to a brand's list by texting a keyword (e.g. "JOIN"). Brands send two kinds of
campaigns: one-time **broadcasts** and automated **flows** (welcome, abandoned-cart, winback).
Store orders (Shopify-style) are attributed back to the send that drove them. Everything is
synthetic and generated by `lab/setup.sql` — no real customer data.

The through-line of the whole deck: **define a business metric once, as a native governed
object, and reuse it identically across Cortex Analyst, Cortex Agents, Cortex Search grounding,
raw SQL, and BI tools.** Every slide should reinforce "one definition, everywhere."

Objects created by `lab/setup.sql` (schema `SMS_MARKETING_DEMO.CORE`):
`DIM_BRAND`, `DIM_SUBSCRIBER`, `DIM_CAMPAIGN`, `FACT_MESSAGE`, `FACT_ORDER`, `SMS_DOC_CHUNKS`,
semantic view `SMS_MARKETING_SV`, Cortex Search service `SMS_DOCS_SEARCH`, agent
`SMS_MARKETING_AGENT`. Warehouse `SMS_MARKETING_WH`.

Live numbers from the generated data (yours will vary slightly by random seed):
flow revenue-per-send ≈ **$59** vs broadcast ≈ **$15**; flow CTR ≈ **13.5%** vs broadcast ≈ **6.8%**;
consent rate ≈ **81–84%** per region; list churn ≈ **9–12%** per region; ~$2M attributed revenue.

## Slide 5: The Semantic View (centerpiece)

**Talking Points:**
- This is the heart of the demo. Walk the DDL top to bottom: tables → relationships → facts → dimensions → metrics. Call out that `revenue_per_send` and `consent_rate` are **derived metrics** composed from base metrics — defined once, reused everywhere.
- Stress it's a *native, schema-level object* with a name, grants, and lineage — not a YAML file sitting on a stage.
- Point at `AI_VERIFIED_QUERIES`: example question→SQL pairs that raise Cortex Analyst accuracy.

**Presenter Notes:**
- Gotcha worth knowing: in DDL, **derived metrics must reference other metrics** (e.g. `DIV0(orders.attributed_revenue, messages.send_count)`), not raw `COUNT(DISTINCT ...)`. That's why the lab defines base metrics first, then composes.
- Privileges to create: `CREATE SEMANTIC VIEW` on the schema + `SELECT` on the base tables. To let a role use it in Analyst: grant `REFERENCES, SELECT` on the view.
- To query it directly: `SELECT * FROM SEMANTIC_VIEW(SMS_MARKETING_SV DIMENSIONS ... METRICS ...)`.

**References:**
- https://docs.snowflake.com/en/sql-reference/sql/create-semantic-view
- https://docs.snowflake.com/en/sql-reference/constructs/semantic_view
- https://docs.snowflake.com/en/user-guide/views-semantic/sql

**Talking Points:**

## Slide 8: Cortex Analyst

**Talking Points:**
- Marketers ask in plain English (five example questions). Accuracy comes from the *view's* metrics, dimensions, and synonyms — plus verified queries — not from prompt engineering.
- Analyst emits governed SQL against `SEMANTIC_VIEW(...)`, so its answers match the dashboard by construction.

**Presenter Notes:**
- Live demo: point Cortex Analyst (or the agent's Analyst tool) at `SMS_MARKETING_SV` and ask "attributed revenue by campaign type" — then "revenue per send for flows vs broadcasts."
- When Analyst is invoked via an agent it uses first-party models only (no open-source LLMs). Enable cross-region inference if a model isn't local: `ALTER ACCOUNT SET CORTEX_ENABLED_CROSS_REGION = 'ANY_REGION';`
- Synonyms matter: "list growth," "opt-in growth," and "new subscribers" all map to the same metric.

## Slide 9: Cortex Search

**Talking Points:**
- Not everything is a number. Campaign briefs, copy, TCPA/consent rules, deliverability, a segmentation playbook, support macros, a quarterly performance review, an attribution whitepaper, and an incident postmortem live in documents.
- Walk the flow: land real PDFs on a stage → `PARSE_DOCUMENT` → build the search service → grounded, cited retrieval. The lab ships 10 real PDFs in `lab/docs/`; `setup.sql` PUTs them to the `@SMS_DOCS` stage and parses them.
- Filterable attributes (`doc_type`, `region`) let the agent scope retrieval.

**Presenter Notes:**
- Real-PDF path is in `setup.sql` (commented): `PUT` files to `@SMS_DOCS`, `ALTER STAGE ... REFRESH`, then `PARSE_DOCUMENT(@SMS_DOCS, RELATIVE_PATH, {'mode':'LAYOUT'})` into `SMS_DOC_CHUNKS`, then rebuild the service.
- The service needs `SNOWFLAKE.CORTEX_USER` and a warehouse for indexing; keep the warehouse MEDIUM or smaller.
- Smoke test with `SNOWFLAKE.CORTEX.SEARCH_PREVIEW(...)`.

## Slide 10: Cortex Search on Call Transcripts

**Talking Points:**
- The second Search corpus is the messiest one: 24 synthetic customer call transcripts (10 support, 8 sales, 6 compliance) — multi-speaker, conversational, no tidy fields. This is where hybrid vector + keyword retrieval clearly beats keyword search.
- Tell the ingestion story: the same corpus, three ways in — (A) chunk directly off the stage with no landing table, (B) `COPY INTO` a durable raw table then chunk (the production ELT path), (C) Snowpipe / Snowpipe Streaming for continuous and real-time loads (narrated in `setup.sql`).
- Attributes `call_type`, `brand`, `call_date` scope retrieval at query time — no re-indexing, RBAC inherited. The RAG closer feeds the top chunks to `AI_COMPLETE` for a cited answer.

**Presenter Notes:**
- `setup.sql` §3b lands `lab/transcripts/*.txt` on `@CALL_TRANSCRIPTS_STAGE`, reassembles whole transcripts, chunks with `SPLIT_TEXT_RECURSIVE_CHARACTER` into `TRANSCRIPT_CHUNKS`, and builds `CALL_TRANSCRIPTS_SEARCH`.
- Demo the attribute filter live: run the compliance-only `SEARCH_PREVIEW` (§7b), then the RAG cell (§7c) or notebook Section 5b.
- The corpus is fully synthetic and client-agnostic (fictional "RelayFox" platform); brand names are invented and used as the `brand` filter.

## Slide 11: Cortex Agent

**Talking Points:**
- The agent has three tools: the semantic view via Analyst (structured KPIs), Marketing_Playbook_Search (our policy/playbook PDFs), and Call_Transcript_Search (what the customer said on the call). Model = auto.

## Slide 12: Layered, Not "Versus"

**Talking Points:**
- Head off the "is this replacing dbt / my BI tool?" question. It isn't. dbt is the code-first system of record for how tables are built; the semantic view is the native governed metric layer; BI tools are the render layer (Omni reads/writes the view bi-directionally).
- Land the three closing cards: **governed by default** (RBAC/RLS/masking/lineage inherited), **open & portable** (OSI YAML in Git, no lock-in), **AI-native** (kills text-to-SQL hallucination).

**Presenter Notes:**
- If a dbt shop pushes back, reinforce: keep authoring in dbt, and *promote* those models/metrics into the semantic view so Analyst and Agents get the same governed definitions.
- "Bi-directional" for Omni means it can read the governed metrics and write back model changes; other BI tools consume the view read-only (e.g. via the Tableau TDS export).

**References:**
- https://docs.snowflake.com/en/user-guide/views-semantic/yaml-vs-ddl
