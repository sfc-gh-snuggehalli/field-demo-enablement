The metric-definition problem

Defined five times

"Attributed revenue" gets re-implemented in the BI tool, the notebook, the ad-hoc SQL, the agent prompt, and the spreadsheet — each subtly different.

Text-to-SQL guesses

Point an LLM at raw tables and it hallucinates joins and filters. Which table? Which date column? Include opted-out subscribers or not?

Numbers don't reconcile

Marketing, finance, and the AI assistant all quote a different "revenue per send." Trust erodes; every meeting relitigates the definition.

Governance bolted on late

Row-level security and column masking live in the warehouse, but the metric layer often sits outside it — so access rules are re-created and drift.

Root cause: the business logic lives everywhere except in one governed place the whole stack can read.

The scenario & the data

An SMS/MMS marketing platform for e-commerce brands

Shoppers opt in via a keyword. Brands send one-off broadcasts and automated flows (welcome, abandoned-cart, winback). Store orders are attributed back to the send that drove them.

Star schema (SQL GENERATOR, 18 months)

DIM_BRAND — store, plan tier, industry
DIM_SUBSCRIBER — opt-in, keyword, channel, region, consent
DIM_CAMPAIGN — broadcast vs flow, theme, send date
FACT_MESSAGE — sends, delivered, clicked, cost
FACT_ORDER — revenue, attributed campaign

7 governed KPIs

Attributed revenue · revenue per send · CTR · opt-in growth · subscriber LTV · list churn rate · consent rate — each defined once in the semantic view and reused by every consumer.

The semantic view: the centerpiece

A native, schema-level, governed object — not a YAML file on a stage

CREATE OR REPLACE SEMANTIC VIEW SMS_MARKETING_SV
  TABLES ( brands, subscribers, campaigns, messages, orders )
  RELATIONSHIPS ( messages(campaign_id) REFERENCES campaigns, ... )
  FACTS ( orders.order_revenue AS revenue, ... )
  DIMENSIONS ( subscribers.region, campaigns.campaign_type, ... )
  METRICS (
    orders.attributed_revenue AS SUM(orders.order_revenue),
    revenue_per_send AS DIV0(orders.attributed_revenue, messages.send_count),
    consent_rate     AS DIV0(subscribers.consented_count, subscribers.subscriber_count)
  )
  AI_VERIFIED_QUERIES ( ... );  -- example Q→SQL pairs raise accuracy
Governed by default: tables, relationships, facts, dimensions, and metrics — plus synonyms, sample values, and verified queries — all live in one object with full RBAC, lineage, and tagging.

Three ways to build it — same object

Programmatic DDL / YAML

CREATE SEMANTIC VIEW in SQL, or a YAML spec via SYSTEM$CREATE_SEMANTIC_VIEW_FROM_YAML. Version it in Git next to your dbt project. Deploy via dbt or Terraform.

CoCo-assisted from dbt

Use the /semantic-views skill to bootstrap the view — or generate it from existing dbt models (schema.yml + metrics). Zero-to-semantic-view in minutes, reusing logic you already wrote.

No-code Snowsight wizard

AI & ML → Cortex Analyst → Semantic View Generator. Pick tables, confirm joins, name metrics, add sample values — all in the UI. Round-trips to YAML/DDL.

Open & portable: all three produce the identical OSI-format object. Export to YAML with SYSTEM$READ_YAML_FROM_SEMANTIC_VIEW and commit it to Git — no lock-in.

Cortex Analyst: natural language, governed SQL

Accuracy comes from the semantic view — not prompt luck

Marketers ask in plain English

• "Attributed revenue by campaign type last quarter"
• "Which region has the fastest opt-in growth?"
• "Revenue per send: flows vs broadcasts"
• "Consent rate and list churn by region"
• "Click-through rate by campaign theme"

Why it's accurate

Analyst maps language to the view's metrics, dimensions, and synonyms — not to raw columns. Verified queries anchor common questions. It emits governed SQL against SEMANTIC_VIEW(...), so results always match the one definition.

No hallucinated joins. The relationships and metric expressions are declared once, so the model never has to guess how messages connects to orders.

Cortex Agent: structured + unstructured, blended

Semantic view (via Analyst) AND Cortex Search as tools; model = auto

"Attributed revenue for the Q3 flash sale came in below plan — what does the incident postmortem say caused the PNW throughput issue, and which campaign brief covered that send?"

Routes the metric to Analyst

Attributed revenue → governed SQL against SMS_MARKETING_SV. Same number as the BI dashboard.

Routes the "why" to Search

Retrieves the PNW throughput postmortem and the Q3 flash-sale brief and quotes them by title — grounded, cited explanation alongside the number.

One assistant, two data shapes. The agent decides which tool answers which part — and both are governed by the objects you already defined.
