A reactive assistant only answers when asked

The user has to already suspect something

Today's in-app assistant waits for a prompt. If a district manager doesn't think to ask about returns at three specific stores, the spike goes unseen for days.

"Scan everything with an LLM" doesn't scale

Feeding the entire data estate to a language model on a schedule is slow and expensive, and most of what it reads is unremarkable noise.

Alerts fire without the "why"

A threshold alert says a number moved. It doesn't say the move is concentrated in Electronics at three Midwest stores, trending since last Tuesday.

Answers must reflect live store data

"I'm visiting Store 1042, what should I focus on?" needs real numbers from this account — not a generic model's best guess.

The goal: an assistant that leads with "return rates at three Midwest stores spiked overnight — here's why, and here are the next steps," and can still answer any ad-hoc question — all while the LLM only ever touches the small slice that already looks interesting.

Pattern: in-warehouse ML functions scan every store on a schedule and precompute a small findings table plus driver attribution. The Cortex Agent orchestrates Analyst, Search, and a custom briefing tool — so the language model reads only the flagged slice, never the full estate.

Step 1 — Detect anomalies without an LLM

SNOWFLAKE.ML.ANOMALY_DETECTION trains one model across every store's return-rate series, then flags outliers on a schedule — pure SQL, no tokens.

-- One multi-series model over all stores; train on history
CREATE OR REPLACE SNOWFLAKE.ML.ANOMALY_DETECTION RETURN_RATE_DETECTOR(
    INPUT_DATA        => TABLE(RETURN_RATE_TRAIN),
    SERIES_COLNAME    => 'store_id',
    TIMESTAMP_COLNAME => 'metric_date',
    TARGET_COLNAME    => 'return_rate',
    LABEL_COLNAME     => '');

-- Detect on the recent window; persist a tiny findings table
CREATE OR REPLACE TABLE STORE_ANOMALY_FINDINGS AS
  SELECT * FROM TABLE(RETURN_RATE_DETECTOR!DETECT_ANOMALIES(
    INPUT_DATA    => TABLE(RETURN_RATE_RECENT),
    SERIES_COLNAME => 'store_id', TIMESTAMP_COLNAME => 'metric_date',
    TARGET_COLNAME => 'return_rate',
    CONFIG_OBJECT => {'prediction_interval': 0.99}));
Proactive engine: a scheduled TASK re-runs DETECT_ANOMALIES every hour, keeping STORE_ANOMALY_FINDINGS fresh. The assistant always has current anomalies to narrate — no LLM was involved in finding them.

Step 2 — Explain the "why" with Top Insights

SNOWFLAKE.ML.TOP_INSIGHTS compares the recent window against the prior baseline and ranks the dimension combinations driving the change.

-- Label the recent window vs. the control period; metric = refund_amount
CREATE OR REPLACE SNOWFLAKE.ML.TOP_INSIGHTS RETURN_DRIVERS_MODEL();

CALL RETURN_DRIVERS_MODEL!GET_DRIVERS(
    INPUT_DATA     => TABLE(RETURN_DRIVERS_INPUT),
    LABEL_COLNAME  => 'label',   -- TRUE = last 10 days
    METRIC_COLNAME => 'metric'); -- refunded dollars
From "a number moved" to a sentence: the drivers come back as product_category = Electronics, store_id in (1003, 1017, 1042), metro = Lakeshore Metro — the raw material for "concentrated in Electronics across three Midwest stores, trending since last Tuesday."

Step 3 — Ad-hoc questions on live store data

A semantic view turns the same tables into a governed model Cortex Analyst can query in natural language — every answer reflects this account's real numbers.

CREATE OR REPLACE SEMANTIC VIEW RETAIL_RETURNS_SV
  TABLES (retailers, stores, returns, store_metrics ...)
  RELATIONSHIPS (returns (store_id) REFERENCES stores, ...)
  METRICS (
    returns.total_refund AS SUM(returns.refund_amount),
    store_metrics.avg_return_rate AS AVG(store_metrics.return_rate));
"I'm visiting Store 1042 — what should I focus on?" Cortex Analyst compiles that into governed SQL over the semantic view. No hallucinated numbers, and the model definitions are reused by every consumer.

The manager asks a follow-up: "Show me refund dollars by category at Store 1042 this month." The agent routes to Cortex Analyst over the semantic view and answers from live data, then can chart it.

Deliver the agent: Snowflake CoWork & Snowflake Intelligence

The same agent object surfaces in Snowflake CoWork / Snowflake Intelligence for interactive use, and is callable from the embedded app over one REST endpoint.

Interactive (CoWork / Intelligence)

Business users chat with PROACTIVE_RETAIL_AGENT in Snowsight → AI & ML → Agents. The agent auto-surfaces its tools; no separate UI to build for internal exploration.

Embedded (REST agent:run)

The in-app assistant calls the agent programmatically and streams the answer back into the product's own chat surface.

-- Same agent, called from the embedded product experience
POST /api/v2/databases/PROACTIVE_RETAIL_DEMO/schemas/ANALYTICS/agents/PROACTIVE_RETAIL_AGENT:run
{ "messages": [ { "role": "user",
    "content": [ { "type": "text",
      "text": "What changed at my stores overnight?" } ] } ] }

Why this stays cheap

Anti-pattern LLM scans everything

Send every store, every day, to a language model on a schedule. Token cost scales with the whole estate; latency grows; most input is unremarkable.

This pattern ML scans, LLM narrates

Set-based ML functions scan every store in-warehouse. The LLM only ever reads the handful of flagged rows and their drivers — token cost scales with what's interesting, not what's stored.

Next Steps

1. Run the lab

Execute setup.sql, then walk the notebook: detect anomalies, explain drivers, query the semantic view, and build the optimized agent.

2. Point it at your data

Swap the sample tables for a real returns / store-metrics feed. The ML functions and semantic view are the only objects that change.

3. Schedule the scan

Resume the monitoring task (or add a Cortex Alert) so findings stay fresh and the briefing is always current.

4. Embed the agent

Wire the product's assistant to the agent:run endpoint, and use CoWork / Intelligence for internal exploration.

The takeaway: detect and explain cheaply in the warehouse; let the language model do what it's best at — narrate and converse — over the small slice that already matters.
