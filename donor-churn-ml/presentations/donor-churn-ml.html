From Fundraising Heuristics to Governed ML

Retention is guesswork

Fundraising teams flag "at-risk" donors with hand-tuned RFM rules in a BI tool. The rules drift, miss silent lapsers, and can't rank a whole file by risk.

Data scientists are boxed out

Real models mean exporting donor data to a separate ML stack — a governance, security, and PII headache that stalls before production.

No path to production

A notebook model is not an MLOps program. Versioning, serving, drift monitoring, and explainability rarely get built, so models decay silently.

Insight stays with analysts

Even a good model produces a score, not an action. Frontline gift officers can't ask "who's at risk in my region and what do I do?"

The gap: a nonprofit fundraising CRM needs churn prediction, MLOps governance, and a natural-language interface — without three separate platforms or moving a single donor record.

1. Feature Store

Define donor features once; serve them point-in-time correct

Register a donor entity and Feature Views for RFM, engagement decay, giving cadence, tenure, channel mix, and wealth signals. The store keeps features fresh and prevents train/serve skew.

from snowflake.ml.feature_store import FeatureStore, Entity, FeatureView

donor = Entity(name="DONOR", join_keys=["DONOR_ID"])
fs.register_entity(donor)

rfm_fv = FeatureView(
    name="DONOR_RFM", entities=[donor],
    feature_df=rfm_df, timestamp_col="AS_OF_TS")   # point-in-time key
fs.register_feature_view(rfm_fv, version="v1")
Why it matters: the same feature definitions power training and inference, so the score a gift officer sees is computed exactly like the model was trained — a governance win for nonprofits handling donor PII.

2. Datasets

Immutable, versioned training snapshots for reproducibility

Materialize a training set from the Feature Store with point-in-time joins on a spine. Each version is a frozen snapshot — reproducible, auditable, and lineage-tracked back to source.

training_ds = fs.generate_dataset(
    name="LAPSE_TRAINING_SET", version="v1",
    spine_df=spine_df,                       # donor_id + as_of_date + label
    features=[rfm_fv, engagement_fv, wealth_fv],
    spine_timestamp_col="AS_OF_TS",
    spine_label_cols=["IS_LAPSED"])
Point-in-time correctness: features are computed as-of the snapshot date and the label comes from the following 12 months — no leakage, and the dataset can be re-materialized identically for any audit or retrain.

Version, govern, and promote models as first-class objects

mv = reg.log_model(

9. ML Observability

Watch drift and performance; alert before the model decays

CREATE MODEL MONITOR DONOR_LAPSE_MONITOR WITH
  MODEL=DONOR_LAPSE_MODEL VERSION='v2' FUNCTION='predict'
  SOURCE=SCORING_LOG BASELINE=TRAINING_SNAPSHOT
  WAREHOUSE=DONOR_CHURN_ML_WH TIMESTAMP_COLUMN=SCORED_AT
  REFRESH_INTERVAL='1 day' AGGREGATION_WINDOW='1 day'
  PREDICTION_SCORE_COLUMNS=('PRED_LAPSE') ACTUAL_CLASS_COLUMNS=('IS_LAPSED')
  SEGMENT_COLUMNS=('REGION');
Governance goal met: track prediction drift, data drift, and accuracy over time — segmented by region — and fire an alert when performance degrades. Query metrics with MODEL_MONITOR_DRIFT_METRIC / _PERFORMANCE_METRIC.
