Pattern: the semantic view governs the numbers (structured), Cortex Search grounds the "why" (unstructured), and the agent orchestrates both tools — so one natural-language question can join a churn metric to the support chats that explain it.

Why a semantic view

Define the business once; every tool inherits it

Governed definitions

Tables, joins, facts, dimensions, and metrics live in one schema-level object with full RBAC and sharing — not copied into each report.

Tool-agnostic

The same view powers Cortex Analyst, agents, and BI tools like Sigma. Change churn once, and it changes everywhere.

A semantic view is the contract between your data and every consumer — human, BI tool, or agent — so "churn rate" means exactly one thing.

Build the semantic view

Tables, relationships, facts, dimensions, and metrics in one statement

CREATE OR REPLACE SEMANTIC VIEW CX_ANALYTICS_SV
  TABLES (
    customers   AS CUSTOMERS   PRIMARY KEY (customer_id),
    subscriptions AS SUBSCRIPTIONS PRIMARY KEY (subscription_id),
    churn       AS CHURN_LABELS PRIMARY KEY (customer_id)
  )
  RELATIONSHIPS (
    subs_to_customers  AS subscriptions (customer_id) REFERENCES customers,
    churn_to_customers AS churn (customer_id) REFERENCES customers
  )
  FACTS ( churn.churned_flag AS IFF(is_churned, 1, 0) )
  METRICS (
    customers.customer_count AS COUNT(customers.customer_id),
    churn.churned_customers  AS SUM(churn.churned_flag),
    churn_rate AS DIV0(churn.churned_customers, customers.customer_count)
  );
churn_rate is a derived metric — a scalar expression of two other metrics — so it's defined once and reused everywhere.

Query it with SEMANTIC_VIEW()

Pick metrics and dimensions; the joins are already defined

-- Churn rate by plan — no JOINs, no metric logic to re-write
SELECT * FROM SEMANTIC_VIEW(
  CX_ANALYTICS_SV
  METRICS churn_rate
  DIMENSIONS customers.plan
);
Analysts select from the view without knowing the physical schema. The definition of churn is guaranteed consistent across every query.

Natural language with Cortex Analyst

Point Analyst at the semantic view for governed NL → SQL

  • Business users ask "What is the churn rate for Enterprise customers in Denver?"
  • Analyst generates SQL against the semantic view, so metrics stay governed.
  • Verified queries (defined in the view) improve accuracy and become onboarding suggestions.
Because Analyst reads the semantic view, it can't invent its own definition of churn — it uses yours.

Unstructured side with Cortex Search

Index the CX chat telemetry from the AI Functions module

CREATE OR REPLACE CORTEX SEARCH SERVICE CHAT_SEARCH
  ON transcript
  ATTRIBUTES customer_id, channel
  WAREHOUSE = CONVERSATIONAL_BI_WH
  TARGET_LAG = '1 hour'
AS SELECT thread_id, transcript, customer_id, channel
   FROM FIELD_CX_DEMO.AI_FUNCTIONS.CHAT_THREADS;
This is the bridge to Module A: the same chat threads scored for sentiment are now searchable by the agent.

One assistant, structured + unstructured tools

→

One answer
CREATE OR REPLACE AGENT CX_INTELLIGENCE_AGENT FROM SPECIFICATION $$
  tools:
    - tool_spec: { type: "cortex_analyst_text_to_sql", name: "CX_Metrics" }
    - tool_spec: { type: "cortex_search", name: "Chat_Telemetry" }
  tool_resources:
    CX_Metrics:     { semantic_view: "FIELD_CX_DEMO.ANALYTICS.CX_ANALYTICS_SV" }
    Chat_Telemetry: { name: "FIELD_CX_DEMO.ANALYTICS.CHAT_SEARCH" }
$$;
Ask "Which churn-risk customers had negative support chats last month?" — the agent uses Analyst for the metric and Search for the conversations, in one answer.

Extend the agent

Custom tools and MCP fit your existing workflow

Custom tools

Add stored procedures / UDFs (type generic) so the agent can trigger actions — e.g. open a save-motion task for an at-risk account.

Data to chart

The built-in data_to_chart tool turns query results into visualizations inside the conversation.

MCP connectors

Connect remote MCP servers so the agent reaches tools like Jira or Salesforce alongside your metrics.

Meets users where they are

The same governed view backs Sigma dashboards and desktop assistants — one definition, many surfaces.
