The customer-experience blind spot

Conversations pile up

A customer-facing GPT assistant and a support line generate thousands of chats and calls a day. Nobody can read them all.

No structured read

Product and CX teams can't answer "what are customers asking about?" or "how do they feel?" without manual sampling.

Churn signals are buried

Frustration and cancellation intent live in free text that never reaches the warehouse or the churn model.

ML is too heavy

Standing up sentiment/topic models, endpoints, and pipelines is weeks of work for a small data team.

The gap: the richest signal about customer experience is unstructured and effectively invisible to analytics.

Sentiment with AI_SENTIMENT

Overall and per-aspect sentiment in one call

-- Overall + category sentiment for each chat thread
SELECT thread_id, customer_id,
  AI_SENTIMENT(
    transcript,
    ['valuation_accuracy', 'pricing_billing', 'onboarding']
  ) AS sentiment
FROM chat_threads;
Returns an OBJECT with an overall record plus one per category, each scored positive / negative / neutral / mixed / unknown. Extract with sentiment:categories.

Topic modeling with AI_CLASSIFY

Map every conversation to a CX taxonomy — single or multi-label

-- Classify each thread into your support topic taxonomy
SELECT thread_id,
  AI_CLASSIFY(
    transcript,
    ['valuation_accuracy', 'pricing_billing', 'onboarding',
     'bug_report', 'cancellation', 'feature_request'],
    {'task_description': 'Support topics for a home-valuation product'}
  ):labels[0]::string AS topic
FROM chat_threads;
Set {'output_mode': 'multi'} when a conversation spans several topics. Add label descriptions and few-shot examples to lift accuracy.

Structured fields with AI_EXTRACT

Pull intent, product area, and requested action out of free text

-- Extract structured fields from each ticket body
SELECT ticket_id,
  AI_EXTRACT(
    text => body,
    responseFormat => {
      'intent': 'What does the customer want to do?',
      'product_area': 'Which product area is mentioned?',
      'urgency': 'Is this urgent? yes or no'
    }
  ):response AS fields
FROM support_tickets;
Ask for a single value per question, in plain English. The same function reads documents on a stage via file => TO_FILE(...).

Theme discovery with AI_AGG & AI_SUMMARIZE_AGG

The "unsupervised" view: surface dominant themes across thousands of rows

-- Top complaints per topic, across the whole corpus
SELECT topic,
  AI_AGG(transcript,
    'Identify the most common customer complaints in these conversations') AS complaints
FROM classified_threads
GROUP BY topic;
These aggregate functions process datasets larger than the model context window and support GROUP BY — ideal for topic discovery, executive summaries, and per-segment rollups.

At-risk detection with AI_FILTER

Keep only the conversations that matter

-- Surface churn-risk / escalation threads in natural language
SELECT thread_id, customer_id, transcript
FROM chat_threads
WHERE AI_FILTER(
  'This customer is frustrated and at risk of cancelling: ' || transcript
);
AI_FILTER returns a BOOLEAN, so it drops straight into a WHERE or CASE. Join the survivors to billing to prioritize saves by revenue.

Same pipeline for voice

Transcribe recordings, then reuse every function above

-- Turn call recordings into text, then score them like chats
SELECT call_id,
  AI_TRANSCRIBE(TO_FILE('@call_recordings', file_name)):text AS transcript
FROM DIRECTORY(@call_recordings);

-- ...then AI_SENTIMENT / AI_CLASSIFY / AI_FILTER on transcript
Chat, in-app, and phone all converge into one CX telemetry table. In this lab, CALL_TRANSCRIPTS ships as text so the pipeline runs without audio files.

Optimize with AI Function Studio

When a built-in isn't enough, build and tune a custom function

Build

Define a custom classification or scoring function on top of AI_COMPLETE for logic the built-ins don't cover (e.g. a custom escalation score).

Evaluate & tune

Compare prompts and models side by side on a labeled sample, review accuracy and cost, and iterate before you productionize.

Note: built-in functions cover most CX telemetry out of the box. Reach for Studio when the customer needs a domain-specific label, a custom rubric, or an accuracy/cost trade-off they want to see measured.

Where AI functions plug in

The same UDFs power the rest of the Cortex stack

Agent custom tool

Register an AI-function UDF as a generic tool in a Cortex Agent spec. The agent's LLM calls it via tool-use when the question needs it (e.g. "how urgent is this message?").

Inside Cortex Analyst

Expose the UDF as a computed column in a semantic view. Analyst's generated SQL invokes the model, so a business user's NL question returns AI-derived labels.

Cortex Search enrichment

AI_EXTRACT and AI_EMBED pre-process documents — structured fields + embeddings — before indexing into a Cortex Search service.

One answer, three tools

An agent can combine Analyst (metrics), Search (relevant tickets), and your custom UDF (urgency) in a single response — all built on AI functions.

Note: AI functions aren't a dead-end SQL feature — they are reusable building blocks. Build the label once as a UDF, then reuse it as an agent tool, an Analyst dimension, and a Search enrichment step. Materialize the column for interactive use so inference runs once per row, not once per question.

Governed metrics with a semantic view

Define churn, revenue, and thumbs-down once; every tool inherits it

-- Churn and app-feedback thumbs-down rate by plan — no JOINs, no re-written logic
SELECT * FROM SEMANTIC_VIEW(
  FIELD_CX_DEMO.ANALYTICS.CX_ANALYTICS_SV
  METRICS churn_rate, thumbs_down_rate, customer_count
  DIMENSIONS customers.plan
);

One definition

Tables, joins, facts, and metrics live in ANALYTICS.CX_ANALYTICS_SV — including thumbs_down_rate fed by the app telemetry. Change it once, it changes everywhere.

Tool-agnostic

The same view powers SEMANTIC_VIEW() SQL, Cortex Analyst NL→SQL, the agent, and BI tools like Sigma.

A semantic view is the contract between your data and every consumer — human, BI tool, or agent — so "churn rate" and "thumbs-down rate" each mean exactly one thing.

What AI Functions actually cost

Token-based billing, and how it runs away on large tables

How you're billed

AI Functions consume credits by tokens processed (input + output), or by pages for document functions — not by warehouse size. A row-wise function over 1M rows makes ~1M model calls.

Where it runs away

Running a row-wise function across a full 1M+ row table without a LIMIT or WHERE, verbose prompts, and oversized models are the usual causes of surprise bills.

-- Track daily credits by function and model (2-5 min latency)
SELECT DATE_TRUNC('day', start_time) AS usage_date,
       function_name, model_name,
       SUM(credits) AS total_credits,
       COUNT(DISTINCT query_id) AS query_count
FROM SNOWFLAKE.ACCOUNT_USAGE.CORTEX_AI_FUNCTIONS_USAGE_HISTORY
WHERE start_time >= DATEADD('day', -30, CURRENT_TIMESTAMP())
GROUP BY 1, 2, 3 ORDER BY usage_date DESC, total_credits DESC;
Best practices: prototype on a LIMIT/sampled subset first • pre-filter rows with WHERE before the AI call • pick the smallest model that passes AI Function Studio eval • keep prompts tight • use aggregates (AI_AGG) instead of row-wise AI_COMPLETE at scale • a warehouse larger than MEDIUM does not speed AI functions up — it just costs more • tag queries with QUERY_TAG for chargeback.

Estimate cost before you run

Price a full-table run without running it

Count the tokens

AI_COUNT_TOKENS('<function>', input) returns the tokens a function will process per row — free to call, never invokes the model. SNOWFLAKE.CORTEX.COUNT_TOKENS('<model>', text) does the same per model.

Tokens → credits → $

Multiply projected tokens by the model's credits-per-million rate (Service Consumption Table), then by your $/credit. Or back out the real rate from CORTEX_FUNCTIONS_QUERY_USAGE_HISTORY and reconcile.

-- Project the cost of AI_SENTIMENT over the whole table, before launching it
SELECT SUM(AI_COUNT_TOKENS('ai_sentiment', transcript)) AS tokens,
       tokens / 1e6 * 0.19  AS est_credits,   -- llama3.1-8b rate
       est_credits * 3.00       AS est_usd       -- your $/credit
FROM CHAT_THREADS;
Why it matters: the same data shows a ~13x cost swing between the smallest and a frontier model — model selection is the biggest lever. Estimate first, pick the smallest model that passes an eval, then scale the run with confidence.

Guardrails: quotas & killing runaway queries

Stop a 1M-row mistake before it drains the budget

Per-user quotas

A first-class quota object enforces monthly/daily per-user credit limits and auto-blocks AI requests at the limit — no custom tasks required.

Spend alerts

An hourly ALERT on CORTEX_AI_FUNCTIONS_USAGE_HISTORY emails admins when account or monthly spend crosses a threshold.

Kill runaway queries

Cancel a specific long-running query on the spot, or automate detection + cancellation with a scheduled task.

Access control

Gate AI Functions behind a dedicated role by revoking CORTEX_USER from PUBLIC, so limits can't be bypassed.

-- Per-user quota that auto-blocks AI spend past the limit
CREATE SNOWFLAKE.CORE.QUOTA my_quota();
CALL my_quota!ADD_SHARED_RESOURCE('AI FUNCTION');
CALL my_quota!SET_PER_USER_LIMIT(120);          -- 120 credits / user / month
CALL my_quota!SET_BLOCK_ENFORCEMENT_ENABLED(TRUE);

-- Kill a runaway query right now
SELECT SYSTEM$CANCEL_QUERY('<query_id>');
Note: lead the cost conversation with quotas — they auto-block at the limit and even terminate in-progress AI calls, so a customer can't burn the budget on one bad query. Alerts and the cancel task are the belt-and-suspenders layer.
