The external-brain tax

A high-volume sales org scores every rep email with AI and mines winning patterns — today via Claude Code over the MCP server. It works, but four costs compound.

Latency on every call

Each tool call is a network round-trip to the external client plus client-side LLM planning — overhead paid on every request, at scale.

Cost blindness

Analyze-everything with a single strong model burns credits fast. No per-user budgets or chargeback when the brain lives outside Snowflake.

Governance gap

Budgets, quotas, tagging, and traces sit outside the data plane. The external client only sees tool outputs; you can't audit what it did.

Connector reliability

OAuth redirect mismatches, DEFAULT_ROLE/warehouse errors, network-policy allow-lists, PrivateLink limits — fragile plumbing to keep alive.

Before: Snowflake-managed MCP server

Wrap existing Snowflake objects as MCP tools; Claude authenticates with built-in OAuth and drives them.

CREATE MCP SERVER GTMAGENTS_MCP FROM SPECIFICATION $$
  tools:
    - name: "gtm-analyst"
      type: "CORTEX_ANALYST_MESSAGE"
      identifier: "GTMAGENTS.DEMO.EMAIL_GTM_SV"
    - name: "framework-search"
      type: "CORTEX_SEARCH_SERVICE_QUERY"
      identifier: "GTMAGENTS.DEMO.FRAMEWORK_SEARCH"
$$;
-- OAuth: no dynamic client registration; redirect URI must match Claude's callback
CREATE SECURITY INTEGRATION GTMAGENTS_MCP_OAUTH
  TYPE=OAUTH OAUTH_CLIENT=CUSTOM ENABLED=TRUE OAUTH_CLIENT_TYPE='CONFIDENTIAL';
Connect gotchas: redirect-URI must match Claude's callback; the session uses the user's DEFAULT_ROLE (set DEFAULT_ROLE + DEFAULT_WAREHOUSE or tool discovery fails); avoid underscores in the account host / DB name in the URL; allow-list the client's outbound IPs if a network policy is on; PrivateLink limits external reach; PAT is the fallback auth.

Targeted analysis: don't score everything

An AI_FILTER gate treats only underperforming / below-threshold emails — the concrete cost lever for a budget-sensitive team.

    'This cold sales email is generic, low-effort, or lacks a clear CTA: {0}', body)))
FROM emails;   -- treat this subset, skip the rest

Wrapped in Snowflake CoWork — no external connector

The supervisor is a standard Cortex Agent, so it appears automatically in CoWork. Business users ask NL questions against governed data with zero connector setup.

External MCP client (before)

Add custom connector, paste URL + client id/secret, complete OAuth, keep redirect URIs and IP allow-lists in sync. Governance lives outside Snowflake.

CoWork (after)

Pick GTM Supervisor from the agent list and ask. No connector, no external auth, full in-plane governance and traceability — same agent, reachable via REST too.

Both worlds stay open: add the supervisor to the MCP server as a CORTEX_AGENT_RUN tool and external clients can still reach it — now governed in-plane.

Measured, not assumed

Native Cortex Agent Evaluation scores GTM_SUPERVISOR against a labeled dataset — the GPA framework: did it answer, did it route, did it reason?

Answer correctness

Does the supervisor's answer match the ground truth for each question? The Goal dimension.

Tool-selection accuracy

Did it route to the right specialist — and call no tool on out-of-scope questions? The Plan/Action dimension.

Logical consistency

Reference-free check that the plan and execution cohere — plus a custom routing-quality judge.

Iterate → promote: improve the orchestration, re-run, compare baseline vs improved, then SET ALIAS = production on the better version — a scheduled Task catches regressions from model or data drift.

When to use which

Use the MCP server when…

You want your existing external client (Claude Code, ChatGPT, Cursor) to reach governed Snowflake tools with minimal setup, and the external brain is where your users already work.

Use Cortex Agents + CoWork when…

You need multi-step orchestration, cost control (cheap + escalate, targeted gating), per-user budgets, server-side traces, and business users asking NL questions with zero connector setup.

Use both when…

Expose the supervisor agent through the MCP server too — one governed object, reachable by external clients and in-plane CoWork alike.
